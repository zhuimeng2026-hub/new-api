# 四模型独立令牌系统设计与开朋结算方案（修订版）

**分析日期**: 2026-06-02
**修订说明**: 开朋网络只提供 1 个 API Key，不可拆分。本方案基于此约束重新设计。

---

## 一、核心约束

| 层级 | 约束 |
|------|------|
| **开朋网络** | 只有 1 个 API Key (`sk-tvyu...`)，不可创建多个 Key，不可按 Key 限模型 |
| **new-api** | 可创建多个 Channel，同一 Key 可填入多个 Channel，靠 Channel ID 区分模型 |
| **keygen** | 可为每个用户创建多个 Token，每个 Token 限制不同模型 |
| **用户** | 每个用户持有 4 个独立 Token，每模型一个，互不干扰 |

## 三、用户侧 Token 设计（keygen 改造）

### 3.1 当前行为（改造前）

```
用户登录 → 选 N 个模型 → 生成 1 个 Token (model_limits = "opus-4-7,gpt-5.5,gpt-img-2,seedance-2-0")
→ 所有模型共用一个 quota 池
```

### 3.2 改造后行为

```
用户登录 → 看到 4 个独立卡片:
  ┌─────────────────────────────────────────────────────────────┐
  │  [Claude Opus 4.7]  [GPT-5.5]  [GPT-Image-2]  [Seedance]  │
  │   额度: 50元         额度: 30元   额度: 20元     额度: 50元  │
  │   [生成Key]         [生成Key]    [生成Key]      [生成Key]   │
  └─────────────────────────────────────────────────────────────┘
→ 每个模型一个独立 Token，独立 quota
```

### 3.3 keygen 后端改造要点

#### 3.3.1 生成逻辑 (`handler/generate.go`)

当前 `GenerateToken` 接收 `models []string`，创建一个包含所有模型的 Token。改造为:

```go
// 新增: 按模型独立生成
type GenerateSingleModelRequest struct {
    Model   string `json:"model" binding:"required"`   // 单个模型名
    Quota   int    `json:"quota" binding:"required"`
    OpenID  string `json:"openid"`
    AffCode string `json:"aff_code"`
}
```

- `ModelLimits` 只包含 1 个模型
- 每个用户对每个模型最多 1 个 Token（检查 openid + model 组合）
- `openid_keys.json` 结构扩展:

```json
{
  "user_openid_123": {
    "opus-4-7":  {"key": "sk-xxx", "base_url": "...", "created_time": 1717, "aff_code": "abc1"},
    "gpt-5.5":   {"key": "sk-yyy", "base_url": "...", "created_time": 1717, "aff_code": "abc1"},
    "gpt-image-2":{"key": "sk-zzz", "base_url": "...", "created_time": 1717, "aff_code": "abc1"},
    "seedance-2-0":{"key": "sk-www", "base_url": "...", "created_time": 1717, "aff_code": "abc1"}
  }
}
```

#### 3.3.2 前端改造 (`frontend/src/`)

- `ModelSidebar.vue` → 改为 4 个独立模型卡片，每个卡片有独立的"生成"按钮和额度选择
- `TokenResult.vue` → 显示 4 个 Key（可折叠/展开）
- `api.ts` → `generateKey()` 改为 `generateKeyForModel(model, quota)`
- 图片/视频功能 → 自动选择对应模型的 Token（从 localStorage 读取）

#### 3.3.3 代理逻辑改造

图片和视频代理需要自动使用对应模型的 Token:

```go
// 当前: 前端传 key
// 改造后: 前端传 openid + model，后端从 openid_keys.json 查找对应 key
func GenerateImage(c *gin.Context) {
    // ... 从 openid_keys.json 查找该用户的 gpt-image-2 token
```

## 五、与开朋的结算方案

### 5.1 核心矛盾

开朋只有一个 Key → 一个额度池 → 无法在开朋侧区分模型。所有成本从同一个账户扣除。

### 5.2 解决方案: new-api 侧独立计量 + 开朋侧总额控制

```
                    new-api 侧 (精确)              开朋侧 (粗粒度)
                    ─────────────────              ─────────────────
按模型计量:         ✅ 4 个 Channel 各有日志       ❌ 只有一个总用量
按模型设预算:       ✅ 每个 Token 独立 quota        ❌ 只有一个总余额
按用户追踪:         ✅ 每个 Token 关联 openid       ❌ 不知道谁用的
成本归因:           ✅ SQL 按 channel_id GROUP BY   ⚠️ 需导出 CSV 手动分拆
```

### 5.3 月度对账流程

```
┌─────────────────────────────────────────────────────────────┐
│                    月度结算流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 每月初: 向开朋充值 (如 2000 RMB)                        │
│     └→ 开朋账户余额增加 (单一额度池)                        │
│                                                             │
│  2. 月中: 系统运行                                          │
│     └→ new-api 按 4 个 Channel ID 分别记录日志              │
│     └→ 每个用户的 Token 独立扣 quota                        │
│                                                             │
│  3. 月末对账:                                               │
│     ┌─────────────────────────────────────────────────┐     │
│     │  new-api 侧 (精确):                             │     │
│     │  SELECT channel_id, model_name,                 │     │
│     │    COUNT(*), SUM(prompt_tokens),                │     │
│     │    SUM(completion_tokens), SUM(quota)           │     │
│     │  FROM logs WHERE channel_id IN (A,B,C,D)        │     │
│     │  GROUP BY channel_id, model_name                │     │
│     ├─────────────────────────────────────────────────┤     │
│     │  开朋侧 (总额):                                 │     │
│     │  导出 CSV → 总 token 数 + 总费用               │     │
│     ├─────────────────────────────────────────────────┤     │
│     │  交叉验证:                                      │     │
│     │  new-api 的 4 个 Channel 用量之和               │     │
│     │  ≈ 开朋 CSV 的总用量                            │     │
│     │  差异 > 5% → 调查                               │     │
│     └─────────────────────────────────────────────────┘     │
│                                                             │
│  4. 利润核算:                                               │
│     └→ 开朋实际成本 (CSV) vs new-api 向用户收取的 quota     │
│     └→ 每个模型独立计算利润率                               │
│     └→ 亏损模型 → 调整 ModelRatio/ModelPrice                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.4 开朋成本估算公式

因为开朋侧无法按模型区分，需要用 new-api 日志**反推开朋成本**:

```sql
-- 按 Channel 估算开朋实际成本
SELECT
  CASE channel_id
    WHEN {claude_channel_id} THEN 'claude-opus-4-7'
    WHEN {gpt55_channel_id}  THEN 'gpt-5.5'
    WHEN {image_channel_id}  THEN 'gpt-image-2'
    WHEN {seed_channel_id}   THEN 'seedance-2-0'
  END as model,
  SUM(prompt_tokens) as input_tok,
  SUM(completion_tokens) as output_tok,
  COUNT(*) as requests,

  -- 开朋成本 (CNY)
  CASE channel_id
    WHEN {claude_channel_id} THEN
      ROUND(SUM(prompt_tokens) * 20.0/1000000
          + SUM(completion_tokens) * 120.0/1000000, 2)
    WHEN {gpt55_channel_id} THEN
      ROUND(SUM(prompt_tokens) * 8.75/1000000
          + SUM(completion_tokens) * 52.5/1000000, 2)
    WHEN {image_channel_id} THEN
      ROUND(COUNT(*) * 0.37, 2)
    WHEN {seed_channel_id} THEN
      ROUND(SUM(prompt_tokens + completion_tokens) * 46.0/1000000, 2)
  END as kapon_cost_cny,

  -- new-api 收入 (CNY)
  ROUND(SUM(quota) * 4.0 / 1000000, 2) as revenue_cny,

  -- 利润率
  ROUND(
    (SUM(quota) * 4.0 / 1000000) /
    NULLIF(
      CASE channel_id
        WHEN {claude_channel_id} THEN SUM(prompt_tokens)*20.0/1000000 + SUM(completion_tokens)*120.0/1000000
        WHEN {gpt55_channel_id}  THEN SUM(prompt_tokens)*8.75/1000000 + SUM(completion_tokens)*52.5/1000000
        WHEN {image_channel_id}  THEN COUNT(*) * 0.37
        WHEN {seed_channel_id}   THEN SUM(prompt_tokens+completion_tokens)*46.0/1000000
      END, 0
    ), 2
  ) as profit_ratio

FROM logs
WHERE channel_id IN ({A}, {B}, {C}, {D})
  AND created_at >= extract(epoch from date_trunc('month', current_date))
GROUP BY channel_id;
```

## 八、架构决策记录

### Q: 为什么不用 1 个 Token 包含 4 个模型?

**A**: 可以，但无法独立控制每个模型的预算。如果用户在 opus-4-7 上烧完了所有 quota，gpt-5.5 也没法用了。独立 Token 让用户可以:
- 给贵模型少充值，便宜模型多充值
- 某个模型用完不影响其他模型
- 管理员可以按模型维度分析用户消费习惯

### Q: 4 个 Channel 共用一个 Key，开朋会不会限流?

**A**: 开朋的限流通常是按账户而非按 Key。4 个 Channel 的请求从开朋视角看是同一个 Key 的流量。如果出现限流，可以:
1. 联系开朋提升 QPS 限额
2. 将低优先级模型的 Channel priority 调低，做排队

### Q: 为什么不直接用开朋的 CSV 做对账?

**A**: 可以用，但开朋 CSV 是**所有模型的混合日志**（因为只有一个 Key）。需要:
1. 从 CSV 按 model_name 过滤
2. 与 new-api 的 channel_id 日志交叉比对
3. 两边的 token 数应该一致，差异来自请求失败/重试
