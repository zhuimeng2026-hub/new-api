# 四模型对账方案：new-api 4 令牌 vs 开朋 1 Key

**核心问题**: new-api 侧有 4 个 Channel（每模型一个），每个用户有 4 个独立 Token；开朋侧只有 1 个 Key、1 个账户。两边的数据结构完全不同，如何对账？

---

## 一、两个数据源的精确结构

### 1.1 数据源 A: new-api logs 表（我方记录）

每条成功的 API 请求产生一条日志（type=2）:

| 字段 | 含义 | 对账用途 |
|------|------|---------|
| `id` | 自增主键 | 去重 |
| `created_at` | UNIX 时间戳 | 按月筛选 |
| `type` | 2=成功消耗, 5=错误(默认不记录) | **只用 type=2** |
| `model_name` | 模型名 (如 `gpt-5.5`) | 按模型分组 |
| `channel_id` | Channel ID | **4 个专用 Channel 各不同** |
| `token_id` | 用户 Token ID | 按用户分组 |
| `token_name` | Token 名 (如 `oversea-0602-0001`) | 可读标识 |
| `prompt_tokens` | 输入 token 数 | 对账核心 |
| `completion_tokens` | 输出 token 数 | 对账核心 |
| `quota` | new-api 内部扣费单位 | 用户侧收入 |
| `other` | JSON (含 image/audio/search 标记) | 区分模态 |
| `use_time` | 请求耗时(秒) | 性能分析 |

**关键**: `type=2` 只记录**成功**的请求。失败请求默认不记录（除非开启 `ERROR_LOG_ENABLED=true`）。

### 1.2 数据源 B: 开朋 CSV 导出（上游记录）

从 `models.kapon.cloud/panel/log` 导出，每行一个请求:

| 字段 | 含义 | 对账用途 |
|------|------|---------|
| Timestamp | 请求时间 | 按月筛选 |
| Model | 模型名 | 按模型分组 |
| PromptToken | 原始输入 token（含缓存） | 验证用 |
| InputToken | 实际计费输入 token | **计费核心** |
| CacheHitToken | 缓存命中 token | 计费项 |
| CacheWriteToken | 缓存写入 token (5min) | 计费项 |
| CacheWrite1hToken | 缓存写入 token (1h) | 计费项 |
| OutputToken | 输出 token (含推理) | **计费核心** |
| ReasoningToken | 推理 token (从 output 中拆出) | 计费项 |
| ImageInputToken | 图片输入 token | gpt-image-2 用 |
| 各 Price 列 | 单价 (CNY/M tokens) | 计算费用 |
| SearchCount | 搜索次数 | Claude 搜索计费 |

**验证关系**: `PromptToken = InputToken + CacheHitToken + CacheWriteToken + CacheWrite1hToken`

### 1.3 两边的关键差异

| 维度 | new-api logs | 开朋 CSV |
|------|-------------|---------|
| **失败请求** | 默认不记录 (type=2 只记成功) | 视频/图片失败不计费; **文本模型未知** |
| **重试** | 每次尝试一条日志 (含失败重试) | 只有最终成功的那次 |
| **Token 粒度** | 只有 `prompt_tokens` + `completion_tokens` | 细分 input/cache_hit/cache_write/output/reasoning |
| **缓存** | 通过 `other` JSON 中的标记间接体现 | 明确拆分 4 种缓存 token |
| **模型名** | new-api 的 model_name | 开朋的 Model (可能大小写不同) |
| **计费单位** | quota (内部单位) | CNY |

## 二、对账公式

### 2.1 第一步: 开朋侧——按模型汇总成本

从 CSV 按 Model 分组，用各模型的核算公式计算每行成本:

#### claude-opus-4-7

```
每行成本 = InputToken  × InputPrice / 1,000,000
         + CacheHitToken × CacheHitPrice / 1,000,000
         + CacheWriteToken × CacheWritePrice / 1,000,000
         + CacheWrite1hToken × CacheWrite1hPrice / 1,000,000
         + OutputToken × OutputPrice / 1,000,000
         + SearchCount × SearchPrice
```

开朋单价 (CNY/M):
- Input: 3.5 (haiku) ~ 20 (opus, 估算)
- Output: 17.5 (haiku) ~ 120 (opus, 估算)
- CacheHit: 0.35 ~ 2
- CacheWrite: 4.375 ~ 25

#### gpt-5.5

```
每行成本 = InputToken × 8.75 / 1,000,000
         + (OutputToken - ReasoningToken) × 52.5 / 1,000,000
         + ReasoningToken × 52.5 / 1,000,000
```
(推理 token 和输出 token 同价)

#### gpt-image-2

```
每行成本 = InputToken × 36 / 1,000,000
         + ImageInputToken × 57.6 / 1,000,000
         + CacheHitToken × 9 / 1,000,000
         + CacheImageToken × 9 / 1,000,000
         + OutputToken × 216 / 1,000,000
```
(OutputToken 占 95%+ 成本，约 0.37 CNY/张)

#### seedance-2-0

```
每行成本 = (InputToken + OutputToken) × 46 / 1,000,000   (720p 无引用)
每行成本 = (InputToken + OutputToken) × 51 / 1,000,000   (1080p 无引用)
每行成本 = (InputToken + OutputToken) × 92 / 1,000,000   (720p 有引用)
每行成本 = (InputToken + OutputToken) × 102 / 1,000,000  (1080p 有引用)
```

**汇总**: `Kapon_总成本 = Σ 所有模型所有行的成本`

### 2.2 第二步: new-api 侧——按 Channel 汇总

```sql
-- 按 4 个专用 Channel 分别汇总
SELECT
  channel_id,
  model_name,
  COUNT(*)                        AS requests,
  SUM(prompt_tokens)              AS total_prompt_tokens,
  SUM(completion_tokens)          AS total_completion_tokens,
  SUM(prompt_tokens + completion_tokens) AS total_tokens,
  SUM(quota)                      AS total_quota
FROM logs
WHERE type = 2                                    -- 只看成功请求
  AND channel_id IN ({ch_opus}, {ch_gpt55}, {ch_image}, {ch_seedance})
  AND created_at >= {month_start_unix}
  AND created_at <  {month_end_unix}
GROUP BY channel_id, model_name
ORDER BY channel_id;
```

### 2.3 第三步: 交叉比对

将两边的数据放在一起:

```
┌──────────────┬──────────────────────────┬──────────────────────────┐
│              │    new-api (我方记录)      │    开朋 CSV (上游记录)     │
│              │    channel_id = X         │    Model = "gpt-5.5"     │
├──────────────┼──────────────────────────┼──────────────────────────┤
│ 请求次数     │ COUNT(*)                 │ CSV 行数                  │
│ 输入 token   │ SUM(prompt_tokens)       │ SUM(InputToken+Cache*)   │
│ 输出 token   │ SUM(completion_tokens)   │ SUM(OutputToken)         │
│ 总 token     │ SUM(prompt+completion)   │ SUM(所有计费 token)       │
│ 费用         │ SUM(quota) → 换算 CNY    │ SUM(每行成本) → CNY       │
└──────────────┴──────────────────────────┴──────────────────────────┘
```

**比对指标**:

```
1. 请求次数匹配:
   newapi_requests ≈ kapon_csv_rows
   差异原因: new-api 重试产生多条日志, 开朋只记最终成功
   容差: ±10% (重试导致 newapi 侧偏多)

2. Token 数匹配:
   newapi_total_tokens ≈ kapon_total_billed_tokens
   差异原因: 缓存 token 计算方式不同
   容差: ±5%

3. 费用匹配:
   newapi_revenue_cny ≈ kapon_cost_cny
   这是最终对账指标
```

## 三、new-api 收入换算

new-api 的 `quota` 是内部单位，换算公式:

```
$1 = 500,000 quota units
CNY = quota / 500,000 × 7.25   (假设汇率 7.25)
```

更精确的逐请求换算:

```
newapi_revenue_cny = Σ (每条日志的 quota / 500000 × 7.25)
```

或者一步到位:

```sql
-- new-api 侧: 按 Channel 汇总，换算 CNY
SELECT
  channel_id,
  model_name,
  COUNT(*) AS requests,
  SUM(prompt_tokens) AS input_tok,
  SUM(completion_tokens) AS output_tok,
  SUM(quota) AS total_quota,
  -- 换算 CNY (quota / 500000 * 7.25)
  ROUND(SUM(quota) * 7.25 / 500000.0, 2) AS revenue_cny
FROM logs
WHERE type = 2
  AND channel_id IN ({A}, {B}, {C}, {D})
  AND created_at >= {month_start}
GROUP BY channel_id, model_name;
```

## 四、对账流程（实操步骤）

### 4.1 每月 1 号执行

```
Step 1: 导出开朋 CSV
  → 登录 models.kapon.cloud/panel/log
  → 筛选上月日期范围
  → 导出 CSV (按 model_name 过滤 4 个模型)

Step 2: 查询 new-api 日志
  → 直接连 PostgreSQL:
     docker exec -it new-api-postgres psql -U root -d new-api
  → 执行第三步的 SQL (替换 channel_id 和时间范围)

Step 3: 比对
  → 将两个结果放在一起
  → 检查: 请求次数、token 总数、费用
  → 标记差异 > 5% 的模型

Step 4: 核算利润
  → 利润 = newapi_revenue_cny - kapon_cost_cny
  → 利润率 = 利润 / kapon_cost_cny
  → 目标: 每个模型 > 1.0x (不亏损)
```

### 4.2 差异排查

| 差异类型 | 可能原因 | 排查方法 |
|---------|---------|---------|
| newapi 请求 > kapon 请求 | new-api 重试了失败请求 | 查 new-api 日志中的重试记录 |
| newapi token < kapon token | 缓存命中: kapon 计费但 newapi prompt_tokens 不含缓存 | 对比 CacheHitToken 列 |
| newapi 费用 > kapon 费用 | new-api 的 ModelRatio 设高了 | 检查 new-api 模型定价 |
| newapi 费用 < kapon 费用 | 缓存/推理 token 未计入 newapi 定价 | 检查 CacheRatio, ReasoningRatio |
| 某模型完全不匹配 | Channel 路由错误，走了 Channel 23 | 检查 channel_id 是否正确 |

### 5.2 月度按 Channel 汇总（对账主查询）

```sql
WITH channel_models AS (
  -- 替换为实际的 channel_id

    (25, 'claude-opus-4-7'),
```

### 5.3 按用户 × 模型汇总（知道谁花了多少）

```sql
SELECT
  l.token_name,
  l.model_name,
  COUNT(*) AS requests,
  SUM(l.prompt_tokens + l.completion_tokens) AS total_tokens,
```

### 5.4 开朋成本估算（用 new-api 数据反推）

```sql
SELECT
  model_name,
  SUM(prompt_tokens) AS input_tok,
  SUM(completion_tokens) AS output_tok,
  COUNT(*) AS requests,

  -- 估算开朋成本 (CNY)
  CASE model_name
    WHEN 'claude-opus-4-7' THEN
      ROUND(
        SUM(prompt_tokens) * 20.0 / 1000000
      + SUM(completion_tokens) * 120.0 / 1000000
      , 2)
    WHEN 'gpt-5.5' THEN
      ROUND(
        SUM(prompt_tokens) * 8.75 / 1000000
      + SUM(completion_tokens) * 52.5 / 1000000
      , 2)
    WHEN 'gpt-image-2' THEN
      ROUND(COUNT(*) * 0.37, 2)   -- ~0.37 CNY/张
    WHEN 'seedance-2-0' THEN
      ROUND(
        SUM(prompt_tokens + completion_tokens) * 46.0 / 1000000
      , 2)  -- 720p 基准
  END AS est_kapon_cost_cny,

  -- new-api 收入 (CNY)
  ROUND(SUM(quota) * 7.25 / 500000.0, 2) AS revenue_cny,

  -- 利润率
  ROUND(
    (SUM(quota) * 7.25 / 500000.0) /
    NULLIF(
      CASE model_name
        WHEN 'claude-opus-4-7' THEN SUM(prompt_tokens)*20.0/1000000 + SUM(completion_tokens)*120.0/1000000
        WHEN 'gpt-5.5'         THEN SUM(prompt_tokens)*8.75/1000000 + SUM(completion_tokens)*52.5/1000000
        WHEN 'gpt-image-2'     THEN COUNT(*) * 0.37
        WHEN 'seedance-2-0'    THEN SUM(prompt_tokens+completion_tokens)*46.0/1000000
      END, 0
    )
  , 2) AS profit_ratio

FROM logs
WHERE type = 2
  AND channel_id IN (25, 26, 27, 28)
  AND created_at >= extract(epoch from timestamp '2026-06-01')
GROUP BY model_name;
```

## 六、已知误差来源与处理

| 误差来源 | 方向 | 幅度 | 处理方式 |
|---------|------|------|---------|
| **new-api 重试** | newapi 请求数 > kapon | +5~15% | newapi 侧偏多是正常的，以 kapon 为准 |
| **缓存 token 粒度** | newapi 只有 prompt_tokens | ±3% | newapi 的 prompt_tokens = InputToken + CacheHitToken (合计)，两边总数应一致 |
| **推理 token** | kapon 拆分 ReasoningToken | 0~5% | gpt-5.5 的推理 token 在 newapi 中计入 completion_tokens，总数一致 |
| **汇率波动** | CNY/USD 变化 | ±3% | 月末用实际汇率重算 |
| **seedance SKU** | 720p vs 1080p 价格差 | 10~55% | CSV 中无法区分 SKU，只能用均价 |
| **gpt-image-2 输出 token** | newapi 的 completion_tokens ≈ OutputToken | 0% | 应完全一致 |

### 最重要的一个对账点

两边的 **token 总数** 应该高度一致（误差 < 5%）:

```
newapi:  SUM(prompt_tokens + completion_tokens) = X
kapon:   SUM(InputToken + CacheHitToken + CacheWriteToken + OutputToken) = Y

X ≈ Y  →  对账通过
X ≠ Y  →  查: 缓存计算差异? 重试? 路由错误?
```

**费用** 不会完全一致，因为 newapi 的 ModelRatio/ModelPrice 是我方定价，不是开朋成本。但两边的 **token 数** 应该一致——这是对账的锚点。
